# TimeKeeper-Observer V1.1
工作学习可用于放置记录专注时间，做事情总得有个盼头

# 一、本版迭代目标总览

本次 1.1 版本需要在保留“记录学习/工作时间 + 高维文明观测”核心框架的前提下，完成以下升级：

1. **初始选择更丰富**：  
   行星类型至少扩展到 6 种；恒星系提供 5 种选择；不同组合应明显影响初始参数、事件倾向和生命诞生节奏。

2. **补完生命诞生前的漫长过程**：  
   从“星球诞生”开始，不应很快跳到“第一个生命”；中间必须经过地壳稳定、原始海洋、大气演变、有机分子富集、前生命化学体系等阶段。

3. **参数驱动而非纯时长驱动**：  
   增加星球参数，如气压、温度、海平面、辐射、水量、氧含量、地质活跃度等；这些参数会随随机事件和重大抉择变化。  
   生命诞生要建立在“条件满足 + 概率触发”的机制上，既要尽量仿真随机性，也不能把概率设得过低，避免体验枯燥。

4. **离线计时逻辑调整**：  
   只有在用户点击“开始观测”后，才开始累计时间；此后无论用户在界面内还是离线，均继续累计；  
   若用户从未点击“开始观测”，则不计入任何时间，页面保持静默，不强行弹出进度。

5. **演化阶段明显增多**：  
   从蛮荒星球到第一个生命，再到物种复杂化，直到智慧生命出现，应拆成更多阶段，并加入更多变量事件来模拟漫长过程。

6. **返回观测站的反馈更丰富**：  
   用户每次返回都必须收到一份普通简报；若本次累计时间跨度超过 1~2 小时，则额外附带一个关键抉择。  
   简报内容要储备更多版式和措辞，尽量避免重复。

7. **观测站需要图案化展示当前状态**：  
   在观测站加入一个能体现“星球现状 / 文明阶段”的图案或徽记，可用大模型生成素材，也可接入公开素材库或直接使用 SVG 程序化生成。

# 二、网站核心定位（v1.1 版）

网站核心仍然是“记录学习/工作时间”，但其反馈方式升级为“高维观测员长期追踪一颗真实演化中的星球”。  
用户不是在“养成一颗会立刻升级的星球”，而是在“持续观察一个拥有长期地质史、生物史、文明史的世界”。

## 2.1 核心体验目标

- 用户开启专注或学习后，现实时间将映射为星球演化时间；
- 星球的演化结果不再只由总时长决定，而是由：
  - 时间积累；
  - 初始星球与恒星系设定；
  - 行星参数变化；
  - 随机事件；
  - 重大抉择；
  共同决定；
- 用户每次返回观测站，都会获得有内容、有变化、有随机感的反馈；
- 星球不是必然线性进化，可能经历环境恶化、演化停滞、灭绝事件、生态复苏等过程；
- 第一个生命、复杂生命、智慧生命的出现都应是“有条件、有概率、但可期待”的事件节点。
